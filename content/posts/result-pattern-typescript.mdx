Error handling in TypeScript often devolves into scattered `try/catch` blocks that obscure control flow. The Result Pattern offers a predictable alternative.

## The problem with exceptions

Exceptions are invisible in type signatures. A function returning `User` might throw at any point, and callers have no way to know without reading the implementation.

## The Result type

```typescript
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };
```

Every operation that can fail returns a `Result`, making failure an explicit part of the API contract.

## Composing results

```typescript
function getUser(id: string): Result<User> {
  if (!id) return { ok: false, error: new Error("ID required") };
  // fetch user...
  return { ok: true, value: user };
}
```

Callers must handle both branches, leading to more robust code paths.

## When to use it

The Result Pattern shines in:

- API route handlers
- Service layer operations
- CLI tools with multiple failure modes
- Any boundary where errors should be explicit
